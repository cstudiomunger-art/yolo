## 一、四大功能实施方案（产出文档，待开发）

依据 5 份设计稿（`00_首页与实用信息中心` / `01_签证板块` / `03_预订留痕` / `04_真人客服 Genius Bar` / `payment_helper_simulator`），结合现有代码库探查，逐功能与产品方对话确认全部实现细节后，输出 `四大功能实施方案_2026-06-15.md`。`02_收藏合并` 本次不纳入。交付方式：四件 + 首页一次性交付。

### 确认要点（对话定稿）
- **首页 IA**：保持克制（问候 + 行程 Hero 含只读签证摘要 + QuickAccess 两格：行前清单 / 实用信息）。实用信息中心独立一页，6 块：Genius Bar(置顶) / 紧急(复用 EmergencyView) / 签证 / 支付 / 交通 / 表达·方言。
- **签证**：纯 Swift `VisaPolicyEngine` 跑 App 端、离线可判，P1–P5 + GATE0 一次做全；后端只 CMS 配数据（`visa_policies × visa_policy_grants` 双表 + `city_visa_tags` + `entry_ports` + `visa_rule_overrides`）；先灌 6 国（UK/US/FR/DE/SG/JP）+ 主要城，其余「问真人确认」兜底；够用静默、不够用弹三路线（R1 纯兴趣 / R2 办 L 签 / R3 签证友好-加城激活过境）；护照有效期可选填；**保留自测口子**（手动选城市仅本地试算、不写回行程）。
- **支付**：流程骨架 + 判定逻辑在 App，文案/建议/短语/外链做 CMS 数据 + JSON 兜底（决策树不进 CMS）；三处境（未出发/已在中国/正卡住）；救援道商家短语用 `AVSpeechSynthesizer` 真离线；卡组织 + 行程类型存 profiles（随 ProfileSync）；绑卡侧门 → Genius Bar `payment_bind`。
- **预订留痕**：扩展 `ItineraryActivity`（kind/hotelId/sourcePlatform）持久化 payload，不建新表；酒店「加入行程」底部 sheet 选 Day；聚合仅在 Plan 行程详情（BookingTraceCard），不进「我的」页；外链购票不留痕。
- **Genius Bar**：自建 Supabase，文字 + 自动翻译 + **图片发送**；**不做推送**（Realtime 仅 App 开着时收）、**不做即时动作验证**；翻译走可后台配置的 `chat-translate` EF；客服仅团队内部账号（`support_agents` + 手动上线）；SOS `priority=emergency` web 置顶；web/support 工作台 clone admin 脚手架 + Realtime，PC 两栏/手机全屏，Cloudflare 部署。
- **交通 / 表达·方言**：新建内容页，CMS 可配 + JSON 兜底；表达·方言音频走后台上传（复用 admin `audio_upload`），方言卡长按全屏大字模式，**不放收藏按钮**。

### 后端
- 新增迁移 062–065：签证引擎 5 表、支付助手 3 表 + profiles 扩列、客服 3 表 + Storage `chat-images`、实用信息 3 表（交通/常用语/方言）。RLS 复用现有私有表/公开内容表两范式。
- admin CMS 注册四组模块（schema.js + nav.js，约 30min/表）；新增 `chat-translate` Edge Function（provider/key/model 后台可配，仿 ai-complete 设置加载）。

### 构建顺序
数据地基（迁移 + CMS + 种子）→ 并行（签证/支付/留痕/内容页/客服后端）→ 集成（web 工作台 + App GeniusBar + 首页 InfoHub + 跨模块 context 跳转）→ 联调验收。
