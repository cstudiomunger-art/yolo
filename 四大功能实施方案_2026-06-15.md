## 2. 签证板块（iOS）

### 2.1 设计要义
两段闭环：**① 检测器**（5 硬输入 → 决策树 → 够用/不够用）+ **② 签证×路线**（Plan 生成路线后三路线选一）。「行程不是输入，地理范围限制是政策的输出」——引擎只收能产生确定判决的硬事实，城市来自 `trip.cities[]`（自测口子时来自临时手动选择）。

### 2.2 数据模型 `YOLO/Models/VisaPolicyModels.swift`
```swift
struct VisaPolicy: Codable { let policyKey, verdict, areaScope: String; let priority: Int; let maxStayDays: Int?; let clockRule: String?; ... }
struct VisaPolicyGrant: Codable { let policyKey, countryCode: String; let effectiveDate, expiryDate: Date?; let maxStayOverride: Int? }
struct CityVisaTag: Codable { let cityId, regionType: String; let isPortOfEntry: Bool }
struct EntryPort: Codable { let portId: String; let transit240h, isHainan: Bool; let cityId: String? }
enum VisaVerdictColor { case green, amber, red }
struct VisaVerdict {
  let color: VisaVerdictColor
  let policyKey: String          // 命中政策
  let maxStayDays: Int?
  let areaScope: String
  let secondaryMatches: [String] // 「你也符合 P3，但 P2 限制更少」
  let blockers: [String]         // 拖累城市
  let latestExitDate: Date?      // 由入境日 + clock_rule 算
}
```

### 2.3 引擎 `YOLO/Services/VisaPolicyEngine.swift`（纯函数，离线）
```
func evaluate(country, departure, entryPort, onward, ticketed, stayDays, entryDate,
              passportValidity?, cities) -> VisaVerdict

# GATE 0（仅当用户填了有效期）
if passportValidity != nil && passportValidity < 6 months: return .red("先换护照")

# 决策树：自上而下，命中限制更少者即停（priority 升序）
P1 互免:   country ∈ grants(P1)                                  → green(协定值, nationwide)
P2 单方面: country ∈ grants(P2) 且 entryDate ∈ [effective,expiry] → green(30, nationwide)
P3 240h:   country ∈ grants(P3) 且 onward 第三国/港澳 且 entryPort.transit240h 且 stay≤240h → amber(240h, transit_ports)
P4 海南:   entryPort.isHainan 且 country ∈ grants(P4)             → amber(hainan)
P5 团体:   随团                                                   → amber(group)
否则:                                                            → red("需办 L 旅游签证")

# 过境第三地判定（与 UI 下拉无关，纯后端逻辑）
onward == 还没定         → 暂不判过境，仅看免签
onward ∈ {香港,澳门}      → 第三地过境 ✓
onward 国/地区 == departure → 往返同国，✗
otherwise               → 第三地联程 ✓

# 逐城复核 → 够用/不够用
for c in cities:
  region = cityVisaTag[c].regionType（叠加 visa_rule_overrides）
  if not 该 region 对 country 免签: blockers += c
verdict.blockers = blockers   # 空=够用，非空=不够用
```
单测：DOC-02 四案例 **SG**(P1 互免)/**FR**(P2 单方面)/**US**(P3 过境)/**IN**(L 兜底) 直接作 fixture。

### 2.4 三路线 `YOLO/Services/VisaTripChecker.swift`（阶段二）
触发：Plan 路线生成完成回调。`v = engine.evaluate(...)`；够用 → 直接展示路线、不打扰；不够用 → 生成三路线：
- **R1 纯兴趣**：原城市（标注代价：需办 L 签）
- **R2 备选**：城市不动 + 办 L 签（给成本与时长）
- **R3 签证友好（推荐）**：找让 blocker 消失的最小改动 ——
  - ① **加城**：能激活过境免签 → 加香港/澳门作第三地出口（P3），体验是加分
  - ② **换城**：blocker 城 → `city_visa_tags.filter(该国免签)` ∩ 内容相似度排序
  - ③ 删城（最后手段）
  - `pick = 候选.sortBy(免签收益 + 体验吸引力).first`

## 3. 支付助手（iOS）

### 3.1 设计要义
「先问处境，不问功能」。三道：**未出发（准备，慢可深）/ 已在中国（压缩）/ 正卡住（救援，秒级离线）**。穷尽信息留后台，用户只走亮着的线。

### 3.2 分层
- **App 内（稳定产品逻辑）**：三处境分流、3 问剪枝（如「只有 Amex → 微信别绑卡」「印度/巴西 → 短信收不到改邮箱」「主去偏远 → 备 2000 现金」）、随身支付卡进度结构。
- **CMS 数据 + JSON 兜底**：所有建议文案（`payment_advice_rules`）、商家短语（`payment_merchant_phrases`）、外链（`payment_helper_links`）。App 内置一份 JSON，CMS 有则覆盖（复用 `ContentCacheStore` 模式）。

### 3.3 数据模型 / 服务
- `YOLO/Models/PaymentHelperModels.swift`：`PaymentAdviceRule`、`MerchantPhrase`、`PaymentHelperLink`。
- `YOLO/Services/PaymentHelperService.swift`：
  - 拉 CMS（失败回落内置 JSON）。
  - 剪枝：`smsAdvice(country)`、`cardAdvice(cards)`、`cashAdvice(trip)`（逻辑对照 `payment_helper_simulator.html` 脚本）。
  - 读写 `UserPreferencesStore.paymentCardTypes / paymentTripType`。

## 5. Genius Bar 真人客服（iOS + web 工作台）

### 5.1 架构
```
web/support 客服端 ──登录/手动上线──> Supabase ──Realtime agents UPDATE──> App grid 绿点
App 用户 ──点选 agent 建会话/发消息──> support_messages ──chat-translate──> body_translated
                                            └──Realtime──> 双端实时同步
```
仅 App 开着时收（无 APNs）。SOS：`priority=emergency`，不指定 agent，web 会话列表置顶、推所有 online 客服抢接。

### 5.2 iOS 数据模型 / 服务

- `YOLO/Services/SupportChatService.swift`（**全项目首个 Realtime 用例**）：
  - 订阅 `support_agents`（grid 在线状态）与当前会话 `support_messages`：`client.channel("conv:<id>").on(.postgresChanges, table:"support_messages") { ... }.subscribe()`。
  - 发文字：插 `support_messages(body_original)` → 调 `chat-translate` 写 `body_translated`。
