# WORKLOG 2026-06-23 · 签证检测页 6 项功能缺陷修复

逐项排查「签证检测」页（`VisaDetectorView`）反馈的功能问题，修掉 6 个真实缺陷：时区错位、结论页重复弹出、口岸编码命名空间不一致、出发/回程国家硬编码、行程来源张冠李戴、国家列表不全。

## 修了什么

1. **时区错位（高）** — `VisaPolicyEngine` 的判定日历从固定 UTC 改为本地时区（`TimeZone.current`）。DatePicker 给的是设备本地墙钟时间，而政策"入境次日 0 时""按日计时"都按本地理解；UTC 日历会把每个日界整体偏移设备时差（如中国 +8h：凌晨 03:00 入境会被算成前一天，导致 grant 生效窗口查错、最晚离境时刻摆错）。`cal`/`dayFormatter` 改为按调用计算，旅客中途跨时区也能跟随。
2. **结论页可能重复弹出（中）** — `VisaDetectorView` 原先用 `.sheet(item:)` 绑定时在 getter 里每次 `IdentifiedRec(id: UUID())` 重建，身份每帧变化会触发 dismiss+重新 present。改为 `@State var verdict: IdentifiedRec?`，仅在 `runEngine` 里构造一次，身份稳定。
3. **口岸编码命名空间不一致（中）** — 口岸选择器用 IATA，而兜底数据里 `cruise_15d` 用 UN/LOCODE（`CNSHA/CNTSN`），二者永不相交 → 邮轮免签的口岸维度永远判不过。兜底口岸改 IATA（`PVG/SHA/TSN`）；选择器补 `TSN 天津滨海`、`TFU 成都天府`，并纠正原 `CTU` 错标"天府"→改"成都双流"。〔注：线上 `visa_policies_v2` 的口岸列需同样使用 IATA，与选择器同命名空间，见下方待办。〕
4. **出发/回程国家硬编码默认（低）** — `departure/onward` 不再写死 `GB`。`departure` 默认跟随护照国籍（`onAppear`），`onward` 默认「还没定」(undecided)。
5. **行程来源张冠李戴（中）** — "正在判定：我的行程"原先无 preset 时直接取全局 `selectedCityIds`（默认 `["beijing"]`），导致**用户没有任何行程也会冒出城市**，且空态兜底永远走不到。新增 `hasRealTrip`（有 Plan preset 或有 `savedItineraries` 才算真实行程）：无真实行程时不再拿 `selectedCityIds` 冒充，标题改"还没有行程"并走空态引导（去 Plan 或手动试算）。
6. **国家列表不全（中）** — 出发/回程国家原为手写 8 国常量。改为复用 onboarding 同源的全量 `passport_countries`（`fetchPassportCountries()`），新增可搜索选择器 `CountrySelectSheet`（flag + 名称 + 搜索，onward 顶部带「还没定」）。退役 `countryOptions/label(for:)/CountryOption`，保留极小 `fallbackCountries` 仅供拉取前/离线兜底。

## 改动清单
- `Services/VisaPolicyEngine.swift`：`cal`/`dayFormatter` 由 UTC 静态常量改为本地时区计算属性（#1）。
- `Services/VisaDataService.swift`：兜底 `cruise_15d` 进出口岸 LOCODE→IATA（#3）。
- `Features/Visa/VisaDetectorView.swift`：
  - `verdict` 状态化 + `.sheet(item:)` 稳定身份（#2）。
  - `departure/onward` 默认值（#4）。
  - `hasRealTrip`/`tripSlugs`/`tripSourceTitle` 行程来源判定 + 空态（#5）。
  - `countries` 全量加载 + `CountrySelectSheet` 可搜索选择器 + `countryLabel`，退役旧国家常量（#6）。
  - 口岸选择器补 `TSN/TFU`、纠正 `CTU` 标注（#3）。

## 追加 · 命名空间对齐（核对线上 + 补邮轮母港）
拉线上 `visa_policies_v2` 实际端口列核对：
- `twov_240h` / `hainan_30d` / `group_asean_xsbn`：端口列已含 IATA（PEK/PVG/HAK/SYX/JHG…），选择器可匹配，**无需改**。
- `cruise_15d`：端口**全为海港 UN/LOCODE**（CNSHA 上海吴淞口、CNTSN 天津、CNXMN 厦门…邮轮母港），选择器一个都没有 → 邮轮免签口岸维度永远判不过。**这是唯一真实缺口**。

根因不是"该用 IATA"——邮轮从海港母港进出，本无机场 IATA，用 LOCODE 正确。对齐方式 = 把邮轮母港补进 `visa_ports`，使选择器命名空间成为政策端口超集。

- **数据库**：`supabase/migrations/079_visa_ports_seaports.sql` — 向 `visa_ports` 补 13 个邮轮母港（code 用政策引用的 UN/LOCODE，display_order 20–32，与机场分段）。
- iOS/admin 无需改：`visa_ports` 是通用维表，补行后选择器自动出现海港选项。
- **离线兜底不动**：bundled `cruise_15d` 用简化 IATA（PVG/SHA/TSN），与 bundled 口岸自洽；线上为准、含海港。

> ⚠️ 待应用：CLI 远端迁移历史为空（078 是 SQL Editor 手动应用，非 `db push`），且本机仅有 anon key（RLS 禁写），故 079 未自动执行。请按 078 同样方式应用 `079_visa_ports_seaports.sql`（SQL Editor 跑一遍即可，幂等 ON CONFLICT）。
> 已应用并复核：`visa_ports` 共 28 行（15 机场 + 13 邮轮母港）。

## 追加 · 政策端口改为从口岸维表勾选（含邮轮口岸可编辑）
政策框架 `visa_policies_v2.entry_ports/exit_ports` 原为 `string_list` 自由文本，需手敲 code、易敲出维表不存在的码而破坏命名空间。改为新增的多选引用类型 `ref_ports_multi`，直接从「口岸维表（IATA）」勾选——海港邮轮母港也在表内，邮轮政策口岸由此可视化编辑。

- `admin/js/ref-cache.js`：refCache 增 `ports`，容错块内拉 `visa_ports`（code/name_zh/display_order/is_active）。
- `admin/js/field-types.js`：新增 `ref_ports_multi`——数组初始化识别 + render case + `renderRefPortsMulti`（checkbox-grid，label「中文名·code」，停用标注；政策引用但不在维表的 code 以 ⚠️ 保留勾选，不静默丢弃）+ 序列化（收集勾选值为数组）。
- `admin/js/schema.js`：`visa_policies_v2` 的 `entry_ports/exit_ports` 由 `string_list` 改 `ref_ports_multi`。
- 端上无需改：仍读 jsonb 数组。

## 追加 · 端到端测试 + 揪出生产级解码 bug（高）
用真实引擎源码 + 线上全量数据建独立测试台 `scripts/visa_live_test.swift`（swiftc 编译 5 个引擎文件，覆盖选择器手动输入 9 例 + 选行程粗判 3 例 + 签证友好路线），跑出一个**潜伏的生产 bug**：

- **`transit_240h` 列永远解码失败 → 整张 `visa_cities` 拉取抛错 → 全量线上数据被丢弃，app 静默回落 6 城兜底**。根因：Supabase 客户端用 `JSONCoding.makeDecoder()`（Foundation `.convertFromSnakeCase`），而 Foundation 会把 `transit_240h` 误转成 `transit240H`（数字后的字母被大写），与属性 `transit240h` 不匹配，`keyNotFound` 抛错。由于 `cities` 与 policies/grants/matrix 在同一 `try await VisaDataSet(...)` 内 await，一处抛错整组回退 → 自 v2 上线（昨日）起，签证引擎一直跑在 bundled 6 城上，从未用过线上 115 城。
- **修复**：`Models/VisaPolicyModels.swift` 给 `VisaCityRow` 写自定义 `init(from:)`/`encode(to:)`，按两种拼写容错解码 `transit240h`（Supabase: `transit240H`；本地缓存用原属性名 `transit240h`），缺失默认 false（该字段仅信息用、引擎不读）。
- **验证**：测试台解码后 `115 cities · 201 grants · 920 matrix · 28 ports` 全部载入；判定矩阵 9/9 通过（GB 单方面/US 240h 过境/SG 互免/IN 红 L 签/GATE0/超期黄牌/无续程票黄牌/US 邮轮随团绿/GB 海南绿）；`xcodebuild` BUILD SUCCEEDED。

> 引擎判定逻辑经线上数据复核正确（绿/黄/红、择优、邮轮随团、海南、240h 过境、GATE0、超期、签证友好路线生成均符合预期）。本次真正的问题在数据解码层，已修。

## 追加 · 判定结果显示「依据政策 · 条目」
结论页原先只显示命中政策中文名 + 一条等宽 debug 行（`policy=… max_stay=… area=…`），且每条政策的 `source_url`（一级信源）客户端从未展示。新增完整政策条目卡，让结论可追溯官方依据。

- `Features/Visa/VisaVerdictView.swift`（仅此一文件，纯展示层）：
  - 新增 `policyEntryCard(_:)`——命中政策的官方名（中/英）+ 关键参数（停留/活动范围/计时，人话）+ 核验日 + **「查看官方公告 →」一级信源链接**（`source_url` → SwiftUI `Link` 走系统浏览器，空则不渲染）。样式复用四维体检单卡片。仅 `!isGate0 && chosenPolicy != nil` 时显示（GATE0 无对应政策不显示；红 visa_L / 黄牌失败政策照常显示其条目）。
  - 移除等宽 debug `metaLine`；`areaLabel`→人话 `areaText`（全国 / 限 N 个指定地区 / 仅口岸限定区）。
  - 按用户选择仅展示命中那一条；备选「你也符合」维持一行文字。
- 验证：`xcodebuild` BUILD SUCCEEDED。线上 8 条政策 `source_url`/`last_verified` 均已填，绿/黄/红（visa_L）均有可点条目卡，GATE0 不显示。
