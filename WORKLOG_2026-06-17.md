# WORKLOG · 2026-06-17

## 客服模块商业化：结束会话 / 历史记录 / 软删除（用户端 + 客服端）

补齐会话生命周期,使 Genius Bar 成为可商用的多用户×多客服系统。决策:**软删除·各删各的**(后台留痕可审计)、**工单模式**(结束=历史只读,再咨询=新会话)、暂保持 3 秒轮询。

### 后端 `068_support_chat_lifecycle.sql`
- `support_conversations` 加 `user_deleted_at` / `agent_deleted_at`(软删标记)+ 用户侧 live 索引。
- 软删/结束都走现有 UPDATE 策略,无需新 RLS;后台(is_admin)读取不过滤 → 全量留痕。

### iOS 用户端
- `SupportConversation` 加 `userDeletedAt/agentDeletedAt` + `isClosed`。
- `SupportChatService`:`loadMyConversations` 改拉 open+closed 且 `user_deleted_at is null`,派生 `activeConversations`/`historyConversations`;新增 `endConversation()`、`deleteMyConversation(_)`;`loadMessages` 加 `.limit(200)`。
- `GeniusBarHomeView`:收件箱分「进行中」「历史会话(只读)」两段,长按 `contextMenu` 删除记录。
- `GeniusBarChatView`:open 时工具栏「结束会话」(确认弹窗);closed 时隐藏输入栏、显示「会话已结束」横幅 +「再次咨询 TA」→ 新建会话。
- `xcodebuild build` EXIT=0。

### web/support 客服端
- 列表查询加 `.is("agent_deleted_at", null)`(排除客服已软删)。
- 查看已结束会话:顶栏「删除记录」(软删 `agent_deleted_at`,后台仍保留)+ 隐藏输入区(只读);未结束仍显示「结束会话」。
- `node --check` 通过。

### admin 后台
- `support_conversations` 只读监控加 `user_deleted_at`/`agent_deleted_at` 字段,运营能看出哪侧删过;admin 读取不受软删过滤(留痕)。

### 验证要点
- 结束 → 进历史只读 → 再咨询新建;用户删历史 → App 消失、客服端/后台仍在;客服删已结束 → 客服端消失、后台仍在;多用户对同一客服 = 多条独立会话。

## 客服翻译：自动翻译开关 + 先发后翻 + 每条消息翻译按钮

规格(对话确认):译成对方语言(用户↔英文/客服↔中文)存库双方可见;**开关=接收端**控制"我收到的消息是否自动翻译",各端各存各的、**默认关**;**先发后翻**(发送即时只带原文,翻译由接收端自动或手动补);每条消息翻译按钮(有译文=显示/隐藏切换,无=翻译,进行中=翻译中…)。

- 迁移 `069_support_message_translate.sql`:`support_messages` 加参与者 UPDATE 策略(此前只 SELECT/INSERT;先发后翻需回写 body_translated)。
- iOS `SupportChatService`:`autoTranslate`(UserDefaults,默认关)、`sendText` 改即时插入不等翻译、`translateMessage(_)`(按 sender 定目标 user→zh/agent→en,回写并重载)、`autoTranslateIncoming`(开关开时翻收到的 agent 消息)、`translatingIds`(翻译中)。
- iOS `GeniusBarChatView`:工具栏自动翻译 Toggle;每条气泡翻译控件(翻译/显示·隐藏译文/翻译中…),译文按 `autoTranslate || 手动显示` 展示。
- web/support:topbar「自动翻译」开关(localStorage 默认关);`sendReply` 即时不等翻译;`translateMessageRow` 按 sender 定目标;`loadMessages` 渲染条件译文 + 翻译按钮 + 开关开时自动翻收到的 user 消息;事件委托处理翻译/显示隐藏。

## 客服翻译：仅翻可见范围（长会话优化）

自动翻译不再进入会话就批量翻全部未翻消息,改为**只翻滚动到视野内的**:
- iOS:去掉 loadMessages 的批量翻译;`GeniusBarChatView` 的 `LazyVStack` 每条 `.onAppear` 时 `shouldAutoTranslate` 才翻(懒渲染=可见范围);开关打开时 `autoTranslateVisibleTail(15)` 只翻当前底部可见的尾部。
- web:loadMessages 给未翻的收到消息 row 打 `data-needtr`;`IntersectionObserver`(root=消息容器)可见才 `translateMessageRow`,译完该行不再标记;3s 重渲染重建 observer。
- 成本有界:超长历史进入只翻屏幕上的几条,向上滚再逐条翻;手动翻译按钮始终可用。`xcodebuild` EXIT=0;web `node --check` 通过。

## 客服：消息改 Realtime 订阅（轮询降为兜底）

把"3 秒轮询"升级为 Supabase Realtime 即时订阅,轮询降到 15 秒只做断网/丢事件兜底。后端表已在 `supabase_realtime` 发布(064),无需新迁移。
- iOS `SupportChatService`:`subscribeRealtime`/`startRealtime`/`stopRealtime` —— `client.channel(...)` 订阅 `support_messages`(insert/update)+ `support_conversations`(update,拿输入态/状态);`subscribeWithError()` 订阅,`realtimeV2.removeChannel` 退订;在 openConversation/startConversation 订阅,stopPolling 退订。pollTask 3s→15s。
  - **防写回环**:`refreshConversationMeta` 去掉 markRead(只读输入态);已读标记移到 `markReadCurrent`(新消息事件/15s 兜底里调),避免"写会话→触发会话事件→再写"死循环。
- web/support:`subscribeListRealtime`(support_conversations + support_messages 变更→重载列表)+ `subscribeMsgRealtime(convId)`(本会话消息→loadMessages;本会话行→refreshActive);`tick` 3s→15s。
- RLS 仍生效(只收到自己可见的行);`xcodebuild` EXIT=0、`subscribe()` 弃用警告已换 `subscribeWithError`;web `node --check` 通过。
