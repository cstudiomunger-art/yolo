## 客服页面卡顿优化(主线程重渲染风暴)
- **诊断**:聊天/首页开着时,主线程(@MainActor)被「全量重载 + 整列表重渲染」反复打满,挤后点击/输入响应。根因:① 自动翻译每条翻完都 `await loadMessages()` 全量重拉,且这次写库又被 Realtime UPDATE 频道收到再拉一次;② `loadMessages`/`loadAgents`/`loadMyConversations`/`refreshConversationMeta`/`recomputeUnread` 每次都整碗重新赋值(Observation 不比较相等性),内容没变也触发重渲染;③ `markRead` 每 15s 轮询+每条新消息都写库 → 触发会话 UPDATE 的 Realtime 回声(刷新会话 + 首页收件箱),自激励空转。
- **修复(仅动 Service,不改交互)**:
  - `loadMessages` / `loadAgents` / `loadMyConversations` / `recomputeUnread` / `refreshConversationMeta`:`if fetched != current { assign }` 守卫,内容没变不赋值、不重渲染(模型已 Hashable)。
  - `translateMessage`:翻译完**本地 patch 单条** `messages[idx]`,不再 `loadMessages()` 全量重拉;Realtime 自我回声被上面的守卫 no-op。
  - `markRead`:DB 写入限流 ~10s/会话(本地未读仍即时清零),掐断「写库→Realtime 回声」循环。
- 点击客服头像进会话仍有必要的网络往返延迟(查重+建会话),但主线程不再被重渲染挤占后,点击明显跟手。
- `xcodebuild build` BUILD SUCCEEDED。

## chat-translate 防刷量(轻量三层)
LLM 翻译接口原本任意登录用户可用任意文本无限调,有刷爆账单风险。加三层防护:
- **迁移 073**:`translate_usage` 计数表 + `translate_rate_check(max_per_min)` 原子 RPC(SECURITY DEFINER):一次调用内完成「资格校验(is_support_agent 或有 open 会话,否则拒)+ 按分钟自增计数 + 是否超限」;另带 `purge_translate_usage()` 供定时清理。
- **函数 chat-translate**:① 输入 `text > 2000 字` 直接返回原文(不调 LLM);② 读 Authorization 用调用者身份调 `translate_rate_check`,不合格/超 30/分钟 → 429(带 translated 原文兜底);③ 通过才调 LLM。
- **两端无需改**:App 端 `translate()` 对非 2xx 抛错→返回 nil→聊天用原文;web 客服端读 `j.translated`,429/超长响应带原文兜底。客服经 is_support_agent 始终合格。
- **待用户**:`supabase db push`(073)+ 重新部署 `chat-translate` 函数。
